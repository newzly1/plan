# 投票功能体验优化 · 设计文档（Spec）

- **日期**：2026-07-11
- **对象**：`webpage/` 投票参考页（6 人印尼选点，CloudBase 实时汇总）
- **前置**：产品仍在作者测试阶段、**未正式发给 6 人**，可放手改，无线上用户顾虑。
- **承接**：`2026-07-07-bali-trip-voting-webpage-design.md`（原始形态）。本文只覆盖“投票体验”的增量优化。

---

## 1. 背景与北极星

页面已能看图/看视频并投票，投票也已接入 CloudBase 实时汇总。但从产品视角看，几处体验拖了“帮 6 人做决定”的后腿：

- **群体共识被藏起来**：云同步的全部价值是“看见大家怎么想”，却埋在浮层深处、要手动刷新。
- **有个死状态**：`无所谓` 记录了值却哪儿都不显示（不进计数、不盖章、不进清单、不进复制、不进汇总）＝纯认知税。
- **没有进度感**：14 个点，用户不知道“我决定了几个”，容易半途而止。
- **身份靠机器**：每台设备一条文档，同一个人换设备/多标签会被当成多个人，`已 N 人参与` 因此不可信。

**北极星**：让这 6 个朋友**尽快、且带着认同感**地敲定行程。投票是手段，“群体达成决定”才是目的。失败态不是“按钮不好按”，而是“没人投完”或“投完了还是看不出集体想要什么”。

**优先级**：
- **P0 · 让群体共识成为主角** —— `群 N/6` 投票进度 + 实时榜单上主表面。
- **P1 · 砍死状态、补进度感** —— `无所谓`→`略过`（计入进度）、`你 N/14` 进度。
- **P2 · 补身份与完成度提醒** —— 首次投票轻提示填名；**用户名即账号**。

## 2. 范围与非目标

**做（本次）**：`去摩擦 + 轻共识` 力度——保留现有页面结构，做下列 ①–⑥。

**不做（YAGNI / 留待日后）**：
- ❌ 每张景点卡内联显示群体票数（更大改动、且引入投票前从众；本次只在 bar/浮层做**聚合**共识）。
- ❌ 真·实时 `.watch()` 推送（用可见性感知轮询替代，更稳更省心）。
- ❌ 真正的密码/账号体系（“用户名即账号、无密码”是刻意选择，见 §8）。
- ❌ 榜单“可砍”负向信号（第三态定为**中性**略过，不加不减）。

## 3. 投票模型：三态都算“一次决定”

| 键 | 内部值 | 榜单权重 | 计入进度 | 卡片钢印 |
|---|---|---|---|---|
| 必去 | `must` | ×2 | ✅ | 绿章「必去」（现有）|
| 可去 | `maybe` | ×1 | ✅ | 墨章「可去」（现有）|
| **略过**（原「无所谓」）| `skip`（原 `no`）| 0（中性）| ✅ | **灰章「略过」（新增，弱化色）** |

- 文案 `无所谓`→`略过`；内部值 `no`→`skip`。
- **`tally.js` 榜单算法不受影响**（本就只数 `must`/`maybe`），但需**为 `skip` 补一条回归测试**：确认 `skip` 被榜单忽略、不计分。
- 「决定」＝三态任一被设置。**进度分母 = `ITEMS.length`（当前 14，随内容自动变化，绝不写死）**。
- 未点任何键 ＝ 未决定，即进度里差的那几个。
- **钢印**：`skip` 也盖章（弱化灰，如 `--ink-faint`），让“这张我处理过了”一眼可见，强化完成度反馈。现有 `must`/`maybe` 钢印不变。

## 4. 底部 bar：从“入口”升级为“状态台”

两行布局，整条仍可点开浮层。承载**你的进度 + 群体投票数 + 群体最爱/定向提醒**。

```
常态（有群体数据、已填名）────────────────────────
│  我的清单 ›                       你 9/14 · 群 4/6 │
│  群体最爱 · A1 乌布   B1 佩尼达   A3 东巴厘          │

有票但还没填名字（第二行做定向提醒）──────────────
│  我的清单 ›                       你 9/14 · 群 4/6 │
│  ⚠ 填个名字才能加入大家的汇总 →                     │

还没人投票 ─────────────────────────────────────
│  我的清单 ›                       你 0/14 · 群 0/6 │
│  还没人投票 · 你可以抢先标记                        │

云同步不可用（SDK 未加载 / 断网）─────────────────
│  我的清单 ›                            你 9/14     │
│  云同步暂不可用 · 可用「一键复制」发群里             │
```

- **你 N/14** ＝ 你决定过的点（must+maybe+skip）/ 总点数（`ITEMS.length`）。
- **群 N/6** ＝ 已投**人数**/固定 6（分母写死 6，见 §8）。
- **群体最爱**：榜单 Top 3 的 `编号 + 中文名`，**静态不滚动**（低干扰）；空间窄则退化到 Top 1。
- 第二行是**状态位**，按优先级择一显示：`云同步不可用` > `有票没名字（提醒填名）` > `还没人投票` > `群体最爱`。
- 点击整条 bar ＝ 打开「我的清单」浮层（现有行为不变）。

## 5. 实时汇总：从“手动刷新”变“自动新鲜”

bar 要常驻显示“群体最爱/群 N/6”，故群体数据必须在**页面加载后就先拉一次**，不能等浮层打开。

- **刷新时机**：① 加载并匿名登录就绪后拉一次；② **页面可见时每 ~45s 轻量轮询**（`visibilityState==='visible'` 才轮询，隐藏即暂停，切回前台立即拉一次）；③ 自己投票后；④ 打开浮层；⑤ 手动「刷新」按钮（保留兜底）。
- 每次拉到数据后，**同时刷新 bar 与（若打开）浮层榜单**。
- 榜单顶部 `已 N 人参与` → 改为 **`群 N/6 已投`**，与 bar 口径统一。
- 轮询成本：CloudBase 只读、6 人量级、45s 间隔 ≈ 可忽略。
- **未来升级点（不在本次）**：真·实时 `.watch()`。

## 6. 首次投票 → 轻提示填名字

- 触发：用户**第一次**给任意点投票（从 0 决定→≥1）、且当前**没填名字**时。
- 形式：**不挡路的 toast**：`给自己起个名字，才能加入大家的汇总 →`，点它直接打开浮层并聚焦名字框（`#nameIn`）。
- 频次：**只提示一次**（本地存 `bali_nudged` 标记），绝不反复打扰，绝不阻断投票。
- 与 §4 呼应：bar 第二行在“有票没名字”时也常驻同义提醒；两者都在说明“名字＝加入群体计数的门票”。

## 7. 身份模型：用户名 = 账号（不再认机器）

**核心变更**：应用层身份从“设备/文档 id”改为“用户输入的名字”。

### 7.1 机制
- **文档 id 由名字推导**：`docId = "u_" + base64url(utf8(normalize(name)))`；`normalize = 去首尾空格 + 内部连续空白折叠为一个空格 + toLowerCase`（拉丁名大小写归一防误分；中文无副作用）。
  - 同一名字在任何设备/浏览器都命中**同一条**文档 → 一个人只算一次。**正是“按用户名判定、不认机器”**。
  - base64url 字符集 `[A-Za-z0-9_-]`，对文档 id 安全；名字上限 16 字符，id 长度可控。
- **云同步以“有名字”为前提**：名字为空 ＝ 只存本机（localStorage + 一键复制），**不进云汇总**。这是“名字即账号”的直接推论，也让 `群 N/6` 永远只数“有名字的人”。
- **匿名登录仍在**，但仅作**写权限门票**；“谁是我”由名字决定。**CloudBase 安全规则不变**：`{ "read": true, "write": "auth.uid != null" }`。
- **turnout**：`群 N/6` 的 N ＝ 集合内文档数 ＝ 不同名字数（每条都有名字，机器重复自然消失）。正常流程下填名发生在**首投之后**（经 §6 提示），故 N 基本等价于“已投人数”；标签用 `群 N/6 已投`。**不加“最多按 6 算”的保险**（作者测试期自行清空 `votes` 集合即可干净）。

### 7.2 关键流程
- **首次填名（本机已有票）**：把当前本机全部投票一次性 upsert 到名字文档 → 立即进入群体汇总。
- **填的名字在云端已存在（换设备续用 / 撞名）→ 采纳并合并（adopt-and-merge，防丢票）**：
  1. 拉取该名字文档；
  2. 对 `union(本机 picks, 云端 picks)` 每个景点：取**优先级更高**者，优先级 `must(3) > maybe(2) > skip(1) > 无(0)`；
  3. `combo`：本机已选则用本机，否则用云端；
  4. 合并结果写回名字文档（set upsert），并**同步回本机 + 重绘**（卡片钢印/进度/清单/榜单）。
  - 效果：换新设备只要输同一名字，先前的票会“拉下来”呈现并续投，不会被本机零星几票覆盖。
- **改名（旧名→新名，均非空）**：先 `remove()` 旧名字文档，再对新名字执行“采纳并合并”。用 `bali_synced_name` 记住“上次写入云端的名字”以定位待删文档。
- **清空名字（非空→空）**：`remove()` 上次同步的名字文档（退出群体汇总），回到只本机。
- **`updated:0 重建` 逻辑作废**：改用 `doc(id).set()` upsert（不存在即建、存在即替换），更简单。旧的 `bali_docid`（设备文档 id）**退役**，改由名字派生 id + `bali_synced_name` 追踪。

### 7.3 平台可行性与兜底（实现时先验证）
- 主路径依赖 CloudBase JS SDK v2 客户端 **自定义 id 的 `doc(id).set()` 创建/替换** 与 `doc(id).remove()`，在现规则 `write: auth.uid != null` 下应可用。**实现第一步先做一次冒烟验证**。
- **兜底**（若平台不允许客户端自定义 id）：改“按名字查找或新建”——`collection.where({name: 归一化名}).get()`，命中则更新其 id、未命中则 `add()`，并把 id 存 `bali_synced_name` 对应的映射。行为语义一致，仅实现细节不同。

## 8. 需接受的固有代价（“名字即账号、无密码”）

- **无密码**：谁输入了你的名字，谁就是那个账号。**利**：换设备/浏览器输同名即续上（配合 §7.2 合并不丢票）；**弊**：6 人里若有人手滑输了别人的名字，会与对方那条合并/覆盖。**熟人私链场景可接受**，本设计按此推进；提示文案可温和引导“各用独特名字”。
- **撞名即同人**：两个真不同的人若用同一名字，会被合并成一条混合账号——同上，靠熟人协调独特名字规避。
- **同名撞车后「改名离开」会串味/重复计数**：若两个真不同的人先都输了同一名字（合并成一条），随后其中一人改名离开，会把合并后的票带走、并删掉原共享文档——汇总可能少算，或（对方再投时）把某景点重复计一次。这是「采纳续投」与「同名即同账号」**同一机制**的固有副作用，无 per-user 身份无法根治（正是无密码的取舍）。**6 人各用不同名字的正常流程不触发**；已知并接受，不为此改数据模型（评审 Task 4 Important，2026-07 记）。

## 12. 风险与缓解

| 风险 | 缓解 |
|---|---|
| 客户端自定义文档 id 的 `set`/`remove` 未被平台允许 | 实现第一步冒烟验证；备好 §7.3“按名字查找或新建”兜底 |
| 换设备/撞名导致覆盖丢票 | §7.2 采纳并合并（union + 优先级），不静默丢票 |
| 无密码被冒名 | 熟人私链场景接受；文案引导独特名字（§8）|
| 轮询多拉数据 | 仅前台轮询、45s 间隔、只读、6 人量级，成本可忽略 |
| 测试期脏数据让 `群 N/6` 虚高 | 正式发人前在 CloudBase 控制台清空 `votes` 集合（作者不加显示保险） |
| 两行 bar 变高吃内容空间 | 作为主 CTA + 状态台的合理代价；`footer` 已留 130px 底部安全区，复核不遮挡 |
